Abstract Factory Pattern

2010-12-03
Software ArchitectureDesign PatternsTypesCreational
Design-Patterns, Tutorials

Design Patterns Overview

Definition

Group Factories That Produce Similar Products. Choose a Factory at Runtime, in Order to Create Different Objects of a Certain Category.

Recipe

  • Definition of the products.
  • In our case, 4 products:
    • product Type A (comes in A1 & A2 Flavors)
    • product Type B (comes in B1 & B2 Flavors)
  • The family of factories.

Notice that when ‘Factory2’ wants to create a product of type ‘A’, it returns ‘A1’ (and not ‘A2’) and this basically tries to emphasize that it is possible to build our factories in a way to return various flavors of available products according to the software needs.

  • Now that we have the factories, let’s have simple static method that chooses a factory for us at runtime to create the right product.
    • The ‘Props’ is an arbitrary object that contains some properties on which the selection is based.

Serve

If you have everything ready, then it would be only a matter of asking ‘FactorySelector’ to choose the right factory and create the desired product.

Important Benefits

  • It basically has advantages similar to the factory pattern.

Example

Let’s say that we want to print some messages.

  • The printer object can be designed to print messages to file, console, paper and etc.
  • Messages can also come in different formats such as unstructured text, xml, json and etc.

As a result, the product hierarchy is define as following.

Two different factories are defined here:

  • ‘Logger’ which logs json messages to file
  • ‘Display’ which shows xml messages in console

Our ‘FactorySelector’ chooses the proper factory based on a given string.

And finally, the client code can use factories as demonstrated.
